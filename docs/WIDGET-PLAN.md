# iOS Home-Screen Widget — Future Workstream

> Reason this isn't in tonight's session: native iOS widgets require a separate Xcode widget extension target, App Group sharing, and Swift/WidgetKit code. None of that fits inside our JS-Expo dev flow on a Windows machine. This doc maps the scope so it can be picked up cleanly when ready.

## What the widget should do (v1)

Single small (2×2 / `systemSmall`) widget showing:

- The user's **top sunny terrace right now** (highest current sun score within 1km of their last-known location)
- Score % (e.g. `82%`)
- Score band label (e.g. `Mostly Sunny`)
- Terrace name (one line, truncated)
- A pin glyph + score-tinted background
- Refresh on tap → opens Zonnie deep-linked to that terrace's detail sheet

Optional medium-size (4×2 `systemMedium`) variant: top-3 list with name + score chip.

## Why bigger isn't worth it for v1

`systemLarge` (4×4) and Lock Screen `accessoryRectangular` are nice-to-haves but each adds testing surface. Coffee in the Sun's widget is a small + medium pair, not a full set.

## Architecture

```
SunBae/                                    ← current Expo app
└── ios/                                   ← generated by `expo prebuild`
    ├── Zonnie/                            ← main app target
    └── ZonnieWidget/                      ← NEW widget extension target
        ├── ZonnieWidget.swift             ← Widget code (WidgetKit)
        ├── ZonnieWidgetBundle.swift
        └── Info.plist
```

Data flow:
1. Main app writes `top-terrace-snapshot.json` to a shared App Group container whenever the user opens the app or the score changes meaningfully (foreground-only — we don't run background tasks).
2. Widget reads from the App Group container on its `TimelineProvider` refresh.
3. WidgetKit refresh budget is conservative (Apple decides when to call us — usually every 15-60 min).

## Required iOS / Apple work

- New widget extension target in Xcode (or via the `expo-apple-targets` config plugin)
- App Group `group.com.spilsbury.zonnie` registered under your Apple Developer account
- Both targets opt into the App Group in the Capabilities panel
- Privacy-sensitive: location used in the main app must be ALSO declared on the widget side if the widget reads location

## Required JS-side work

- A small `widgets/snapshot.ts` module that serialises the current top-terrace state (id, name, lat, lng, score, scoreBand, ts) to a JSON file in the App Group container path
- Hook `app/_layout.tsx` to write the snapshot:
  - On launch (after favorites + weather hydrate)
  - On significant time-window or score change (debounce ~5s)
- Deep-link handling: when the widget is tapped, iOS opens `zonnie://terrace/<id>` — already mostly wired since we use `expo-linking`, just needs a route at `app/terrace/[id].tsx` that dispatches `useSelectionStore.select(id)` and routes back to the map

## Required native (Swift / WidgetKit) work

- `TimelineProvider` reading the snapshot file
- `View` body rendering the score badge + name + glyph
- A few sizes (`systemSmall` minimum, `systemMedium` optional)
- Dynamic Island / Live Activity is OUT of scope for v1

## What this looks like to ship

1. Run `npx expo prebuild --platform ios` once (generates the `ios/` folder; not committed by default)
2. Add the widget extension target via the `expo-apple-targets` config plugin (declarative — not the manual Xcode dance)
3. Write the Swift code (~150 lines)
4. Write the JS snapshot writer (~50 lines)
5. Test in iOS Simulator on a Mac (no Mac = blocked here) OR via EAS Build + install on device
6. EAS Build a new dev/preview build with the widget enabled

## Effort estimate

- Setup + Apple Developer config: 1 hr (one-time)
- Native widget code: 2-3 hr (new territory in our project)
- JS snapshot pipeline: 1-2 hr
- Testing + iteration: 2-4 hr
- **Total: 1-2 days of focused work**, gated on either:
  - Access to a Mac for iOS Simulator-driven dev, OR
  - EAS Build cycles (~20 min each) used as the test loop

## Why not a simpler approach (PWA shortcut, Live Activity, etc.)

- **Lock Screen widget** (`accessoryInline` / `accessoryCircular` / `accessoryRectangular`): same WidgetKit infrastructure as Home Screen, no shortcut.
- **Live Activity**: only triggers from an active session (e.g. while driving to a terrace). Not an "always-on score" use case.
- **App Clip**: completely different feature. Doesn't help here.
- **Web push / notification widgets**: Apple doesn't allow third-party apps to render arbitrary content on Home Screen except via WidgetKit extensions.
